# L1/L1.5 Realtime Briefing and Adaptive Visuals Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Upgrade the pre-meeting briefing so Sarah can use GPT Realtime voice plus typed fallback to ask Neo4j-backed questions, while the same web interface displays the best visual representation of queried client memory.

**Architecture:** Keep the existing Next.js App Router and Neo4j memory helpers. Add a normalized memory query response shape, make `/api/memory/query` return adaptive answer/display payloads, and teach the Realtime briefing component to call that backend both for typed fallback and Realtime function-tool queries. The UI keeps fixed context panels plus an adaptive answer panel for cards, tables, graph, timeline, recommendations, citations, and missing-info/proactive next steps.

**Tech Stack:** Next.js 16 App Router, React 19, TypeScript, Tailwind CSS, Neo4j driver, OpenAI Realtime WebRTC.

---

## Confirmed product decisions

- V1 user is Sarah only.
- L1 supports all briefing styles: 30-second summary, 2-minute deeper context, and interactive advisor Q&A.
- Q&A uses GPT Realtime and can query backend/Neo4j via tools.
- Missing memory should be explicit and proactive: say what is missing, suggest how to collect it, and propose a pending action/memory gap.
- Browser voice and typed fallback are both required.
- L1.5 uses recommended hybrid UI: fixed context dashboard plus adaptive answer/result view.
- Visual display should include memory cards, relationship graph, timeline, action/follow-up table, referral recommendation card, and evidence snippets. Skip charts/trends until there is real time-series data.
- Query display selection should be hybrid/rule-based: backend returns structured intent/display mode; UI maps it to stable components.
- Keep Mr. Tan as the single primary demo client for L1/L1.5.
- During pre-meeting Q&A, advisor notes should become pending action/memory-gap suggestions, not permanent memory.

## Normalized response shape

Add a type like:

```ts
export type MemoryDisplayMode =
  | "brief"
  | "cards"
  | "table"
  | "graph"
  | "timeline"
  | "recommendation"
  | "missing_info";

export type EvidenceSnippet = {
  id: string;
  label: string;
  source: string;
  snippet: string;
  confidence?: number;
};

export type MemoryQueryVisualResponse = {
  clientId: string;
  query: string;
  source: "neo4j" | "demo";
  displayMode: MemoryDisplayMode;
  answer: string;
  citations: EvidenceSnippet[];
  cards?: Array<{ id: string; title: string; eyebrow: string; body: string; meta?: string }>;
  rows?: Array<{ label: string; value: string; detail?: string }>;
  graph?: { nodes: GraphNode[]; edges: GraphEdge[] };
  actions?: SuggestedAction[];
  missingInfo?: { title: string; reason: string; suggestedNextStep: string };
  warning?: string;
};
```

## Implementation tasks

### Task 1: Add response types

**Objective:** Define normalized L1/L1.5 query response types.

**Files:**
- Modify: `lib/types.ts`

**Steps:**
1. Add `MemoryDisplayMode`, `EvidenceSnippet`, `SuggestedAction`, and `MemoryQueryVisualResponse` exports.
2. Keep existing types backward-compatible.
3. Run `npm run lint`.

### Task 2: Build adaptive response formatter

**Objective:** Convert `ClientContext` plus a user query into a stable visual answer payload.

**Files:**
- Create: `lib/memory-query-response.ts`

**Steps:**
1. Add intent detection for keywords:
   - referral/introduce/specialist/lawyer -> `recommendation`
   - relationship/family/network/who -> `graph`
   - promise/follow-up/action/send/remind -> `table`
   - last time/history/timeline/discuss -> `timeline`
   - brief/summary/know -> `brief` or `cards`
   - unknown/no matching evidence -> `missing_info`
2. Generate concise `answer` text grounded in `ClientContext`.
3. Generate `citations` from matching memories/actions with source snippets.
4. Include `graph`, `rows`, `cards`, and `actions` only when relevant.
5. Add missing-info fallback with proactive next step.

### Task 3: Update memory query API

**Objective:** Make `/api/memory/query` return normalized visual responses.

**Files:**
- Modify: `app/api/memory/query/route.ts`
- Modify: `lib/neo4j-memory.ts` if helper is needed

**Steps:**
1. Keep `clientId` and `query` inputs.
2. Fetch `ClientContext` via `getClientContextWithMemoryLayer(clientId)`.
3. Build adaptive response from context and query.
4. Include existing `queryClientMemory` raw result only if useful as `rawResults` or omit to keep API clean.
5. Return JSON matching `MemoryQueryVisualResponse`.

### Task 6: Add Realtime function-call loop

**Objective:** Let GPT Realtime query Neo4j-backed memory during voice Q&A.

**Files:**
- Modify: `components/voice-briefing.tsx`

**Steps:**
1. Pass a `query_client_memory` tool definition when calling `/api/realtime/token`.
2. Update `buildRealtimeInstructions` to tell the model to call `query_client_memory` for client-specific questions.
3. In `handleRealtimeEvent`, detect function-call completion events such as `response.function_call_arguments.done` or completed function-call output items.
4. Parse arguments `{ query: string }`.
5. Call `/api/memory/query` with `context.client.id` and query.
6. Send `conversation.item.create` with `type: "function_call_output"`, `call_id`, and JSON output.
7. Send a follow-up `response.create` instructing the model to answer from tool output and mention missing-info next steps when relevant.

## Risks / tradeoffs

- Realtime function-call event names can change; implement tolerant handlers for multiple event shapes.
- Browser Realtime voice still requires a valid `OPENAI_API_KEY`; typed fallback must remain useful without it.
- Avoid committing `.env.local` or any API key.
- Do not add charts until there is real trend data.
